## 需求列表

### R-001 硬件交互接口预留（项目级）
- 背景/痛点：作为 GUI 界面生成器，项目需在产品层面预留统一的硬件交互接口，使界面元素能绑定“设备交互能力”并获得一致反馈，而不聚焦具体设备的控制细节。
- 用户场景：在设计器或预览中，体验者为按钮、开关等组件绑定通用的设备交互（例如：触发动作、查询状态、订阅事件），以便后续与任意设备映射。
- 功能要求：
  - 提供通用的设备交互能力类型：动作（触发）、查询（获取状态）、事件（接收通知）。
  - 在设备不可用或未就绪时，界面给出明确提示或模拟反馈，不中断主要流程。
  - 全程离线可用；交互过程支持取消与重试；错误信息中文且不遮挡操作。
  - 绑定目标可为设备、设备组或占位符（后续再映射到具体设备）。
- 交互规则：
  - 触发后显示进行中与最终结果（成功/失败），反馈样式统一、可辨识。
  - 支持单个与批量目标的选择与触发（如项目存在多设备）。
- 约束与边界：不限定设备品牌、类型与参数模型；不描述具体控制行为，仅要求具备统一绑定与反馈能力。
- 验收标准：
  - 体验者能在界面为元素绑定并触发一类设备交互，并获得清晰的结果提示。
  - 无实际设备时不崩溃，提供模拟或说明性反馈。
  - 离线可用，文案清晰，不暴露实现术语。
- 优先级：P2
- 状态：待评审

### R-002 画布交互：单击选中即可拖动
- 背景/痛点：当前需要“点击一下再移动”，不符合直觉，拖动效率低。
- 用户场景：在画布中对组件进行微调或布局调整。
- 功能要求：
  - 单击组件即选中；按住不松即可直接拖动，无需额外步骤。
  - 支持多选（如按住 Shift）与框选；支持取消选择（点击空白或按 Esc）。
  - 拖动过程中有明确的视觉反馈（选中态、对齐线、网格吸附如开启）。
  - 提供设置项，可关闭“单击即拖动”以兼容旧习惯。
- 交互规则：
  - 选中对象显示明显边框或手柄；拖动时位置变化实时可见。
  - 框选与多选时，反馈一致且易辨识。
- 约束与边界：不要求在本需求中处理旋转/变形，专注于移动体验。
- 验收标准：
  - 体验者无需切换到“移动”工具即可完成拖动操作。
  - 多选、框选、取消选择均工作正常，反馈清晰。
  - 拖动定位准确，对齐反馈正常。
- 优先级：P1
- 状态：待评审

### R-003 资源库统一管理与属性选择器
- 背景/痛点：属性面板需要直接填“图片路径”，易错且不便，无法预览与管理。
- 用户场景：体验者导入图片/字体等资源，在属性面板通过选择器引用。
- 功能要求：
  - 资源库面板：支持导入、预览、搜索、删除，资源以中文名称展示。
  - 属性选择器：在属性面板点击按钮弹出资源选择器，无需手输路径。
  - 支持分组/标签与显示引用状态；可识别未引用资源并清理。
  - 导出或构建时，资源被完整打包到项目资产目录，引用不丢失。
- 交互规则：
  - 导入后立即可预览缩略图与基础信息；选择后属性自动填充。
  - 清理操作需确认提示，避免误删在用资源。
- 约束与边界：本需求不限定资源文件的具体存储结构，仅关注用户导入、选择与清理体验。
- 验收标准：
  - 体验者能导入图片并在属性面板通过选择器完成引用。
  - 导出/构建结果包含所需资源，界面展示正常。
  - 可列出未引用资源并一键清理，过程有确认与结果反馈。
- 优先级：P1
- 状态：待评审

### R-007 WebAssembly 真实预览集成
- 背景/痛点：当前设计器使用 React 模拟控件渲染，对于复杂控件（3D、Video、Canvas 绘图等）无法准确呈现真实效果，预览与实际运行存在差异。
- 用户场景：
  - 设计包含 3D 模型、视频播放、自定义绘图等复杂控件的界面时，需要在设计器中看到真实效果。
  - 调整控件属性后，希望立即看到与最终运行一致的渲染结果。
- 功能要求：
  - 将 HoneyGUI C/C++ SDK 编译为 WebAssembly 模块。
  - 在设计器预览中加载 Wasm 模块，直接运行真实的 HoneyGUI 渲染代码。
  - 支持 OpenGL ES → WebGL 的自动转换（通过 Emscripten）。
  - 提供降级方案：Wasm 加载失败时回退到 React 模拟渲染。
  - 保持现有的快速预览体验，Wasm 渲染延迟不超过 500ms。
- 交互规则：
  - 预览模式切换：提供"模拟预览"和"真实预览"两种模式。
  - 首次加载 Wasm 模块时显示加载进度。
  - 复杂控件在"模拟预览"下显示占位符，在"真实预览"下显示实际效果。
- 约束与边界：
  - 第一阶段仅支持渲染预览，不支持交互事件（点击、拖拽等）。
  - 不要求所有控件都用 Wasm 渲染，简单控件可继续使用 React 模拟。
  - Wasm 模块大小需优化，避免影响插件加载速度。
- 验收标准：
  - 能够在设计器中看到 3D 模型、视频播放等复杂控件的真实渲染效果。
  - 预览效果与编译仿真结果一致。
  - Wasm 加载失败时能正常降级到 React 模拟渲染。
  - 文档提供 Emscripten 编译配置和集成指南。
- 优先级：P2（长期优化）
- 状态：待评审

## 已完成需求（2025-11-23）

### R-004 拖拽组件到正确容器 ✅
- 背景/痛点：拖拽组件到第二个view时，组件总是被添加到第一个view，无法准确放置。
- 用户场景：在画布中有多个容器（view）时，需要将组件拖拽到指定的容器中。
- 功能要求：
  - 根据鼠标释放位置准确判断目标容器。
  - 支持嵌套容器的正确识别（选择最内层容器）。
  - 拖拽过程中高亮显示目标容器。
  - 如果鼠标位置不在任何容器内，给出明确提示。
- 修复方案：使用 `e.currentTarget` 替代 `e.target` 获取画布容器坐标。
- 验收标准：✅ 拖拽到第二个view正确添加，坐标计算准确。
- 优先级：P0（严重BUG）
- 状态：已完成
- 提交：2855038

### R-005 hg_view默认大小使用项目分辨率 ✅
- 背景/痛点：新建hg_view时大小固定为350x250，与项目配置的分辨率不一致。
- 用户场景：创建新的hg_view容器时，希望默认大小与项目分辨率匹配。
- 功能要求：
  - 新建hg_view时，自动使用项目配置的分辨率作为默认大小。
  - 如果没有项目配置，回退到默认值350x250。
  - 其他组件不受影响。
- 修复方案：在创建组件时检查 `componentType === 'hg_view'`，使用 `canvasSize` 作为默认大小。
- 验收标准：✅ 项目分辨率800x480时，新建hg_view大小为800x480。
- 优先级：P1
- 状态：已完成
- 提交：d17b397

### R-006 前端调试日志可查看 ✅
- 背景/痛点：前端LOG无法查看，调试困难，无法定位问题。
- 用户场景：开发调试时需要查看前端的console.log输出。
- 功能要求：
  - 开发模式下保留所有console.log。
  - 提供DEBUG开关控制详细日志。
  - 支持通过VSCode开发者工具查看前端LOG。
  - Webpack配置支持保留日志的生产构建。
- 修复方案：
  - 添加 `DEBUG_DROP` 开关（App.tsx）
  - Webpack配置 `drop_console: false`
  - 使用生产模式构建但保留日志
- 验收标准：✅ 能够查看前端console.log，DEBUG开关工作正常。
- 优先级：P0（开发必需）
- 状态：已完成
- 提交：52a8e8a, 46f22bb
