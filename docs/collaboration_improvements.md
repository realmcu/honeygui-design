# 协同开发功能改进计划

本文档记录了针对离线局域网协同开发功能的潜在改进建议与技术演进方向。

## 1. 冲突解决 (Conflict Resolution)

当前状态：主要依赖"最后写入者胜"（Last Write Wins）策略，在低延迟局域网环境下通常表现良好，但在高并发场景下可能导致状态不一致。

**改进建议：**
- **引入 CRDT (Conflict-free Replicated Data Type)**：
  - 使用 Yjs 或 Automerge 等库来管理共享文档状态。
  - 确保无论操作顺序如何，最终文档状态在所有客户端上保持一致。
- **操作转换 (Operational Transformation, OT)**：
  - 作为 CRDT 的替代方案，适用于需要中央服务器协调的场景（当前架构已有 Host 角色，可行性较高）。

## 2. 连接稳定性与重连机制

当前状态：已实现基础的连接超时检测，但缺乏自动重连机制。

**改进建议：**
- **自动重连策略**：
  - 实现指数退避（Exponential Backoff）重连算法。
  - 当 WebSocket 意外断开时，Guest 端应尝试自动重新连接 Host。
- **断线重传**：
  - 维护一个未确认操作的消息队列。
  - 连接恢复后，自动发送断线期间产生的本地操作。

## 3. 安全性增强

当前状态：局域网明文通信，无鉴权机制。

**改进建议：**
- **访问控制**：
  - **PIN 码/访问令牌**：Host 启动时生成一个随机 PIN 码，Guest 连接时必须输入该 PIN 码才能加入会话。
  - **连接审批**：Host 端可以收到新连接请求的通知，并选择接受或拒绝。
- **通信加密**：
  - 虽然是在局域网内，但在条件允许的情况下，探索使用 WSS (WebSocket Secure) 或对消息体进行简单的加密处理。

## 4. 用户体验优化

**改进建议：**
- **光标协同**：
  - 显示其他用户的光标位置和选中状态，增强即时感知。
- **用户列表与状态**：
  - 在侧边栏或状态栏显示当前在线用户列表。
  - 显示用户正在编辑的组件或区域。

## 5. 架构演进

**改进建议：**
- **Diff 同步优化**：
  - 优化全量同步与增量同步的策略，减少大文件协同时的网络带宽占用。
  - 使用 JSON Patch 或类似的 Diff 标准来描述变更。
